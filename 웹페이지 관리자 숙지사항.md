# 웹페이지 관리자 숙지사항

배포·DB·회원·데이터 정합성과 관련해, 운영 시 꼭 알아두면 좋은 사항을 번호로 누적합니다. 새로운 주의사항이 생기면 아래 목록 끝에 이어서 번호를 붙여 추가합니다.

---

1. **학부모–자녀 연결과 학생 삭제 (`parent_profiles`)**  
   `parent_profiles.linked_student_id`는 자녀의 `profiles.id`를 가리키며, 외래키 제약은 `ON DELETE RESTRICT`입니다. 자녀 학생 계정(`profiles` 행)을 삭제하려면 **먼저 해당 학생에 연결된 학부모 계정을 해제·삭제**하거나, 연결이 없는 상태로 만든 뒤 삭제해야 합니다. 그렇지 않으면 DB에서 학생 삭제가 거절됩니다.

2. **학부모 기능 DB 마이그레이션**  
   학부모·수업반 관련 테이블은 코드 저장소의 `db/parent_schema.sql`에 정의되어 있습니다. **Supabase SQL Editor 등에서 이 스크립트를 실행한 뒤**에야 앱의 학부모 API가 정상 동작합니다. 실행 전 백업·스테이징 검증을 권장합니다.

3. **학부모 계정과 자녀 수 (현재 설계)**  
   한 학부모(`parent_profiles`)는 **연결된 자녀 학생 UUID 하나(`linked_student_id`)**만 갖는 MVP 구조입니다. 형제·자매까지 한 아이디로 묶으려면 추후 `parent_student_links` 같은 별도 연결 테이블 설계가 필요합니다.

4. **학부모 가입 신청의 아이디 중복**  
   `parent_signup_requests`에서는 **`status = '대기'`인 신청끼리만** `username` 중복이 막힙니다. 이미 거절·승인 처리된 옛 행과 같은 아이디로 다시 신청하는 것은 가능합니다.

5. **수업반 이름**  
   `class_groups.name`에는 DB 레벨에서 **고유(UNIQUE) 제약을 두지 않았습니다.** 같은 이름의 반이 학기별로 둘 이상 생길 수 있으므로, 관리 화면에서는 가능하면 **이름 + 설명** 또는 내부 구분으로 혼동을 줄이는 것이 좋습니다.

6. **학부모 요청 유형**  
   `parent_requests.request_type`은 학생용 `student_requests`와 동일하게 `보강영상`, `질문`, `상담`을 허용합니다. 학부모 화면에서 일부 유형만 쓰더라도 DB 제약은 위 세 가지입니다. 정책 변경 시 마이그레이션과 API 검증을 함께 수정해야 합니다.

7. **RLS와 API 접근**  
   학부모 관련 테이블에는 RLS가 켜져 있고, 별도 anon 정책은 없습니다. **서버에서 `SUPABASE_SERVICE_ROLE_KEY`로 접근하는 API**는 정상 동작하며, 클라이언트에서 anon 키로 직접 테이블을 조회하는 방식은 사용하지 않는 것이 안전합니다.

8. **학부모 전용 경로와 쿠키**  
   `/parent`, `/parent/...` 는 미들웨어에서 **`parent_auth=true`** 와 **UUID 형식의 `parent_id`** 쿠키가 모두 있어야 통과합니다. `/parent-login`, `/auth/parent-signup` 은 matcher에 넣지 않아 **로그인·가입 페이지는 차단되지 않습니다.** 미들웨어는 쿠키 형식만 확인하며, 계정 승인 여부·실제 존재는 각 API에서 `getParentSession`으로 DB에 다시 확인합니다. 로그아웃 시에는 `parent_auth`, `parent_id` 쿠키를 함께 제거해야 합니다.

9. **학부모 가입 신청 아이디 저장 방식**  
   API(`/api/auth/parent-signup`)는 아이디를 **앞뒤 공백 제거 후 소문자로 정규화**해 `parent_signup_requests.username`에 저장합니다. 대소문자만 다른 중복 신청은 막히며, 승인 후 로그인 시에도 동일 규칙으로 맞춰야 합니다.

10. **학부모 비밀번호 저장**  
   `parent_profiles.password_hash`에는 **bcrypt** 해시만 저장합니다(코드에서는 `bcryptjs`, 라운드 10). 가입 신청 테이블의 평문 비밀번호는 관리자가 승인할 때 해시로 바꿔 넣어야 하며, DB에 평문을 직접 넣으면 로그인 `compare`가 실패합니다.

11. **학부모 가입 승인 시 학생 연결**  
   관리자 화면에서 학부모 신청을 승인할 때 **반드시 `profiles.id`(승인된 학생) 하나를 선택**해야 합니다. 미승인 학생은 API에서 연결할 수 없습니다. 신청에 적힌 자녀 이름과 **완전히 동일한 이름의 승인 학생이 한 명뿐**이면 드롭다운에 자동 반영됩니다. 동명이인·철자 차이가 있으면 수동으로 올바른 학생을 고릅니다.

12. **수업반 삭제와 연쇄 삭제**  
   `class_groups` 행을 삭제하면 DB 제약에 따라 **`class_group_students`(반 소속)** 와 **`class_reports`(주간 리포트)** 가 함께 삭제됩니다. 삭제 전에 백업이 필요하면 관리자 화면에서 내용을 복사해 두거나 DB 백업을 사용하세요.

13. **학부모 전용 API**  
   학부모 로그인 쿠키가 있을 때만 사용하는 경로입니다. `GET /api/parent/profile`(본인·자녀 요약), `GET /api/parent/exam-records`(자녀 성적 읽기 전용), `GET /api/parent/class-report`(자녀가 속한 반의 리포트), `GET|POST|DELETE /api/parent/requests`(학부모 문의·소프트 삭제). 학생용 `/api/requests`·`/api/exam-records`와 테이블이 다르므로 혼동하지 마세요.

14. **학부모 로그아웃**  
   `POST /api/auth/parent-logout`은 `parent_auth`·`parent_id` 쿠키를 제거합니다. 학부모 페이지 상단 「로그아웃」에서 호출한 뒤 `/parent-login`으로 이동합니다.

15. **미들웨어(middleware)와 보호 경로**  
   학생 앱 화면 `/`, `/video`, `/material`, `/request`, **`/mypage`** 는 `student_auth` 쿠키(또는 관리자 `admin_auth`)가 없으면 `/login`으로 보냅니다. **`/parent`** 는 `parent_auth`+`parent_id`가 없으면 `/parent-login`으로 보냅니다. `/login`, `/parent-login`, `/auth/signup`, `/auth/parent-signup` 등은 matcher에 포함되지 않아 **비로그인 접근이 가능**합니다. 새로운 “로그인 필수” 페이지를 만들면 `middleware.ts`의 `matcher`에 경로를 추가해야 동일하게 보호됩니다.
